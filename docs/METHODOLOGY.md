# Benchmarking Methodology

## 가설

> 동일한 의미의 프롬프트를 영어/한국어/일본어로 입력했을 때, Claude의 응답 품질에 통계적으로 유의미한 차이가 있다.

## 실험 설계

### 독립 변수
- **프롬프트 언어**: English (en), Korean (ko), Japanese (ja)
- **벤치마크 모드**: native (원문 그대로), translated (번역 후 전송)

### 종속 변수
5개 평가 차원의 점수 (1-5 스케일):
1. 구현정확도 (Implementation Accuracy)
2. 의도파악 (Intent Comprehension)
3. 할루시네이션 (Hallucination)
4. 코드버그 발생도 (Code Bug Rate)
5. 누락/아웃데이트 빈번도 (Omission/Outdated Frequency)

### 통제 변수
- 동일 모델 (claude-sonnet-4-6)
- temperature = 0.0 (재현성)
- 동일 max_tokens
- 동일 의미의 프롬프트 (전문 번역)

## 테스트 케이스 설계 원칙

1. **의미 동등성**: 각 언어의 프롬프트는 정확히 같은 의미를 전달
2. **자연스러움**: 기계 번역이 아닌, 해당 언어 화자가 자연스럽게 작성할 표현 사용
3. **기술 용어 보존**: 고유명사, API명, 라이브러리명은 원문 유지
4. **난이도 분포**: Easy 30%, Medium 40%, Hard 30%
5. **카테고리 균형**: 4개 카테고리에 동일 10개씩

## 채점 방법

### 자동 채점 (Automated Scoring)
- 코드 실행 테스트 → 구현정확도, 코드버그 점수
- pass rate 매핑: 100%→5, 75%→4, 50%→3, 25%→2, 0%→1

### LLM-as-Judge 채점
- Claude가 영어로 응답을 평가 (언어 바이어스 제거)
- 3회 평가 후 평균, 분산 > 1.0이면 4차 평가 추가
- 구조화된 JSON 출력으로 일관성 확보

### 언어 바이어스 제거
- Judge는 항상 영어 원문 프롬프트 기준으로 평가
- 응답의 "내용"만 평가, 언어 자체는 평가하지 않음

## 통계 분석

### 검정 방법
- **주 검정**: Paired t-test (동일 테스트 케이스 비교)
- **보조 검정**: Wilcoxon signed-rank (비모수적)
- **다중 비교 보정**: Bonferroni (5개 차원 × 언어 쌍 수)
- **효과 크기**: Cohen's d

### 유의 수준
- α = 0.05 (Bonferroni 보정 후)
- 95% 신뢰구간

### 시행 횟수
- 테스트 케이스당 3회 시행 (LLM 비결정성 고려)
- 총 API 호출: 40 cases × 3 languages × 3 trials = 360

## 한계점

1. **테스트 케이스 수**: 40개는 통계적 검정력에 제한이 있을 수 있음
2. **번역 품질**: 프롬프트 번역의 품질이 결과에 영향
3. **LLM-as-Judge 한계**: 판정자 모델 자체의 언어 편향 가능성
4. **시간적 변동**: API 모델 업데이트에 따른 결과 변동
5. **단일 모델**: Claude만 테스트 (다른 LLM과의 비교 없음)
